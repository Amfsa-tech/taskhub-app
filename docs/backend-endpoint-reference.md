# TaskHub — Screen → Backend Endpoint Reference

> Generated by scanning every screen under `app/` plus the shared `components/taskhub/*` action modals and the `lib/`, `context/` layers.
> Purpose: walk this list against the **backend repo** to confirm which endpoints exist and which still need to be built to make each screen functional.

## Current state in one paragraph

Only the **auth** domain is wired to a real backend. The API client (`lib/api/client.ts`) is a generic typed `fetch` wrapper (`api.get/post/put/patch/delete`) that injects a bearer token, and `lib/auth/auth-api.ts` binds the `/api/auth/*` endpoints. **Everything else — tasks, bids, taskers, wallet/payments, messages, notifications, reviews, verification, support, blocked users, sessions — currently runs on hardcoded mock data, local component state, or `AsyncStorage` via `TaskContext` / `LocationContext`.** Location search is fully functional through the **external Geoapify API** (no TaskHub backend involved).

### Conventions (from `lib/`)

- **Base URL:** `https://task-hub-backend.onrender.com` (override with `EXPO_PUBLIC_API_URL`) — `lib/config.ts`
- **Auth header:** `Authorization: Bearer <token>` (token persisted in SecureStore) — `lib/api/client.ts`
- **Response envelope:** `{ status: 'success' | 'error', message, ...data }` — `lib/api/types.ts`
- **Account types:** `user` vs `tasker` — many endpoints have parallel `/user` and `/tasker` variants.

### Status legend (used throughout)

| Mark | Meaning |
|---|---|
| ✅ | Endpoint is bound **and called** in the app today |
| 🟡 | Endpoint is **defined in app code** (`lib/auth`) but the screen doesn't call it yet — backend almost certainly already has it; this is a **frontend wiring gap** |
| 🔴 | **Not in the app at all** — the screen uses mock/local data. Verify whether the backend provides it; build if missing |
| 🌐 | **External** service (Geoapify) — no TaskHub backend needed |

## Part 4 — Notes for the backend cross-check

1. **Auth is the baseline.** The 11 `/api/auth/*` endpoints in `lib/auth/auth-api.ts` should already exist on the backend. Several auth screens (`create-account`, `otp`, `forgot-password`, `create-new-password`) don't call them yet — that's **frontend wiring**, not missing backend, so don't be misled by the UI.
2. **The single most important new endpoint is `POST /api/tasks`** (create task). It's the terminal step of both the manual post flow (`post-review`) and the voice/AI flow (`review-task` / `voice-confirm`), and today it only writes to `AsyncStorage` via `TaskContext.addTask`.
3. **Escrow/payments span several screens** (`task-agreement` → `track-task` → `receipt`, plus `chat` and `wallet`). Confirm the backend's escrow model and whether payment is `POST /api/tasks/:id/payment` vs a generic `POST /api/payments` — the app currently implies both; pick one and standardize.
4. **`user` vs `tasker`.** Profile, stats, and some lists likely need both variants (mirroring `/api/auth/user` and `/api/auth/tasker`). Confirm the backend's convention (path variant vs role claim in the token).
5. **Paths here are inferred, RESTful guesses.** Where the backend already names things differently, treat this doc as the *capability* checklist (does the feature have an endpoint?) and adapt paths in `lib/` accordingly. The recommended next step is to add domain API modules next to `lib/auth/auth-api.ts` (e.g. `lib/tasks/tasks-api.ts`, `lib/wallet/…`) as endpoints are confirmed.
6. **Starter-template leftovers.** `app/(tabs)/_layout.tsx`, `app/(tabs)/index.tsx`, `app/(tabs)/explore.tsx`, and `app/modal.tsx` are unmodified Expo starter files (not part of the TaskHub product) — no endpoints; consider removing.
